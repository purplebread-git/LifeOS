# Architecture Review

Системный обзор ядра после завершения подсистемы Document Ingestion
(PR #18–#24). Цель — не список задач, а фиксация выводов, которые через полгода
объясняют, *почему* архитектура выглядит так, а не иначе. Обзор основан на
фактическом коде, а не на планах.

## Принцип: чем оправдан интерфейс

Абстракция (ABC) в проекте оправдана **одной из двух независимых причин** —
достаточно любой:

1. **Вариативность** — есть 2+ реальные реализации, отличающиеся поведением.
2. **Инверсия зависимости (DIP)** — интерфейс не даёт высокоуровневому модулю
   зависеть от низкоуровневого, даже при единственной реализации.

Следствие: «одна реализация» — **не** признак лишней абстракции. Лишняя
абстракция — та, у которой нет ни (1), ни (2): нет второй реализации и она
ничего не развязывает. Только такие — кандидаты на удаление.

Обобщать «по форме» (два похожих класса) по-прежнему нельзя — обобщаем по
устойчивой доменной модели и правилу трёх реальных потребителей.

Более общий процессный принцип (Phase 1–2):

**Новая инфраструктура появляется только после нескольких независимых
подтверждений одной и той же гипотезы.**

Следствия на практике: нет BaseExtractor, GenericRanker, GenericMatch,
PluginLoader, универсального `register(...)` — каждый раз не хватало
достаточного числа независимых подтверждений.

## Принцип: куда подключаются расширения платформы

**Composition Root — единственная точка подключения платформенных расширений.**

Если для нового Plugin нужно менять Agent, ConversationEngine, ToolManager или
другую центральную часть ядра (а не только `container.py` и сам Plugin),
модель расширения нарушена. Подтверждено тремя осями: Tool, ContextLayer,
DocumentExtractor. Аудит API: `docs/plugin-api-review.md`. PluginLoader —
только при новых фактах.

## Статус абстракций (по фактам кода)

Оправданы вариативностью (2+ реализации):

| Абстракция | Реализации |
|---|---|
| `Tool` | 6 инструментов |
| `MemoryProvider` | InMemory / Sqlite / SemanticSqlite |
| `KnowledgeProvider` | InMemory / Sqlite / SemanticSqlite |
| `DocumentExtractor` | PlainText / Markdown / Pdf |
| `ContextLayer` | System / Memory / Knowledge / ConversationHistory |
| `LLMProvider` | OpenAI / Null |
| `ConversationEngine` | Simple / Tool |
| `ContextBuilder` | Layered / Simple |

Оправданы инверсией зависимости (пока одна реализация, но развязывают):

| Абстракция | Что развязывает |
|---|---|
| `MemoryRanker` / `KnowledgeRanker` | политика ранжирования отделена от storage — `SemanticSqlite*Provider` зависит от ranker, а не наоборот |
| `EmbeddingProvider` | семантический провайдер не привязан к OpenAI |
| `Chunker` | ingestion не знает стратегию нарезки |
| `ExtractorRegistry` (не ABC, но точка расширения) | сервис не знает, как выбирается extractor |
| `ToolManager` | движок/агент не знают набор инструментов |
| `ConversationRepository` | агент не знает способ хранения |

Наблюдать (одна реализация, ближе к спекулятивным): `Agent`.
`PluginManager` / `PluginRegistry` после #27–#30 — подтверждённый runtime
(+ две оси расширения); детали — `docs/plugin-api-review.md`.

Оговорка про `ContextLayer`: реализаций четыре, но **все они producer'ы**
(`return [*context, *mine]`). Дополнительная обобщённость `apply(...)` —
трансформация всего контекста, ради которой выбран pipeline вместо producer —
ещё не использована ни одним слоем. Она подтвердится с приходом
`TokenBudget` / `Trimming`, а пока это осознанный аванс.

## Дублирование Memory ↔ Knowledge

Зеркальность подсистем намеренная. Различия реальны и оправдывают раздельное
существование:

* `MemoryProvider` — CRUD (`add/get/update/delete/search`);
  `KnowledgeProvider` — batch + управление источником
  (`add_batch/search/list_sources/delete_source`);
* `MemoryEntry.created_at` (recency) vs `KnowledgeChunk.source` (attribution) —
  эволюция полей расходится.

Крупнейшее буквальное дублирование — в `SemanticSqlite*Provider`
(сбор кандидатов, save-always, деградация к substring, `_safe_embed`,
`_to_record`/`_to_entry`). Плюс дубли ranker'ов, `*Match` + `MatchType`.

Вывод: объединять доменные модели/провайдеры **рано** (два потребителя, честная
дивергенция). Слияние создало бы ложную связанность между подсистемами.

Watch-item №1: если появится **третий** семантический store (другой vector DB),
это станет сигналом извлечь общий helper семантического retrieval — но не раньше.

## ExecutionContext

Здоров. Несёт только capabilities (`memory`, `knowledge`, `ingestion`) +
идентичность (`conversation_id`, `user_id`) + `metadata`. Технических
зависимостей (`database`, `settings`, `embedding_provider`, `logger`) нет —
service locator'ом не стал. Точки наблюдения: `metadata: dict[str, Any]`
(потенциальная «свалка») и пока не используемый `user_id`.

## DI Container

Читается сверху вниз в порядке зависимостей; комментарии объясняют «почему»;
ядро чистое (`app/core` не импортирует `dependency-injector`). Косметика на
будущее: инструменты создаются как обычные объекты в списке (единственное
отступление от `providers.Singleton`); при дальнейшем росте — под-контейнеры
(Memory / Knowledge / Ingestion) через композицию контейнеров.

## Найденный дефект: корреляция tool_call_id

Единственная действительно критичная проблема аудита — и она про корректность
протокола, а не про перестройку архитектуры.

Инструменты жёстко проставляли `ToolResult.tool_call_id` = собственное имя
(`"remember"`), значение утекало в TOOL-сообщение и в OpenAI, где `tool_call_id`
обязан совпадать с `id` из `tool_calls[]` ассистента (`call_xxx`). На реальном
OpenAI это ломало многошаговый ReAct-цикл и параллельные tool-calls; тесты не
ловили из-за фейковых LLM.

Причина — неверная ответственность: инструмент знает только
`arguments → content` и не видит, каким был вызов. Корреляцию должен выполнять
`ToolManager`, у которого есть `ToolCall.id`. Исправлено в PR #25:
`tool_call_id` стал необязательным и не устанавливается инструментами, менеджер
штампует `ToolCall.id` на результат.

## Что сознательно НЕ вводится

По фактам аудита оснований нет: `BaseExtractor`, `ExtractorError`,
`ResolutionStrategy`, обобщённый `Ranker[T]` / `Match[T]`, слияние
Memory/Knowledge. Мелкие дубли (`_last_user_message` / `_extract_text` в
context-слоях, `_as_utc` в memory-провайдерах) оставлены в статусе «наблюдаем»:
несколько строк, вынести всегда успеем; кандидат — при появлении третьего
реального потребителя.

## Итог

Почти каждая ключевая абстракция прошла проверку реальным расширением, а не
осталась предсказанием. Аудит выявил единственный критичный дефект (корреляция
tool_call_id), касающийся взаимодействия с API, а не устройства ядра, — признак
того, что архитектура зрелая и накопленного долга практически нет.
